1. One token becomes top-k expert rows

CODE Let T be tokens, H hidden width, F expert FFN width, E experts, k selected experts per token, W the full process-group size, and P=W/EP the FFN tensor-parallel size. The example uses T=1024, H=4096, F=14336, E=8, k=2.

splits[e]

How many expanded rows are assigned to expert e. Its sum is T × k.

scatter_index

Where an expanded token/expert choice lands in expert-grouped order.

routing_idx

Layer 1’s inverse lookup: which expert-output row contributes to each token/top-k slot.

inputs_shard = rand([ntokens / tp_size, h])

intermediate_output = zeros([nrows_ep, ffn_size / ffn_tp_size])

Execution. Routing metadata is made once for the example; layer 0 expands and groups rows, while layer 1 uses inverse routing to sum the selected expert results.

Ownership / stream. Indices and activations are device tensors used on the caller’s CUDA stream.

Invariant. sum(splits) = T×k; scatter_index and the layer-1 routing lookup must describe the same expansion.

Why this exists. Grouped GEMM wants expert-contiguous problems even though Python starts with token-contiguous input.

2. Tensor ownership under TP and EP

CODE Sequence-parallel input means each rank in the full group initially owns [T/W,H]. All-gather reconstructs [T,H]. EP assigns each rank a subset of experts, while FFN tensor parallelism of size P=W/EP shards F.

Where each quantity comes from. E, H, and F belong to the MoE model configuration; Hopper versus Ampere does not choose them. The distributed run supplies W and configures EP. This implementation then derives P=W/EP, E_local=E/EP, and F_local=F/P. Crucially, E_local counts expert identities represented on a rank. 3. Grouped GEMM, tiles, CTAs, and SM pressure

PAPER An MoE expert batch is not one rectangular GEMM: each expert has a different M from splits, but shares K and N. A grouped GEMM represents those expert problems together. A tile scheduler maps M×N tiles to CTAs; an SM runs resident CTAs subject to register/shared-memory limits.

INFERENCE “More CTAs” is not automatically more overlap. A communication kernel that occupies every SM can delay GEMM; too few communication resources can expose network latency. COMET’s resource assignment is therefore part of the algorithm.

Oversubscription in one sentence. There are usually more logical tiles than can be resident; persistent or strided CTA loops reuse a bounded set of blocks, which makes scheduling order—not just tile count—observable.

4. CUDA ordering is not global visibility

Stream order

Commands in one stream execute in issue order. Different streams need an event, semaphore wait/write, or another dependency.

CTA barrier

__syncthreads() coordinates threads of one CTA. It does not by itself publish bytes to a host-enqueued transfer.

System scope

__threadfence_system() and system-scope release operations make publication meaningful beyond the issuing GPU’s ordinary device scope.

The patch uses three distinct ideas and they must not be collapsed:

  1. a kernel finishes writes to a staging slot;
  2. it makes those writes system-visible, then release-publishes a flag;
  3. an auxiliary stream waits on the flag before issuing NVSHMEM RMA.
src/moe_gather_rs/topk_gather_rs_v2.cu:667–675 · DIFF · abridgedGPU kernel · system publication
if (params.nnodes > 1 && g != params.node_idx) {
  __threadfence_system();
  __syncthreads();
  if (threadIdx.x == 0) {
    int done = atomicAdd(params.group_counters + g * params.n_split + sid, 1) + 1;
    if (done == gridDim.x)
      atomic_store_release_sys(params.group_flags + g * params.n_split + sid, 1);
  }
}

Execution. Every block completes its share; the last counter increment publishes the group flag.

Ownership / stream. GPU CTAs write symmetric staging_send; flags/counters live in real device allocations and are watched by internode_stream.

Invariant. Observing flag ≥ 1 must imply all bytes in that (owner-node, N-split) slot are system-visible.

Why this exists. A host-enqueued put must never race unfinished producer stores.

5. Node-contiguous rank topology

CODE DistEnv assumes equal GPUs per node and contiguous global ranks:

local_world_size = world_size / nnodes node_idx = global_rank / local_world_size local_rank = global_rank % local_world_size global_rank(node, local) = node × local_world_size + local
include/flux/utils.h:52–96 · HEAD · abridgedHost/device POD topology
struct DistEnv {
  int32_t rank, world_size, nnodes;
  int32_t local_world_size, local_rank, node_idx;

  global_rank_to_node_idx_local_rank(int global_rank) const {
    divmod_local_world_size.fast_divmod(node_idx, local_rank, global_rank);
    return {node_idx, local_rank};
  }
  int local_rank_to_global_rank(int local_rank, int node_idx) const {
    return node_idx * local_world_size + local_rank;
  }
};

Execution. Fast div/mod is usable in host orchestration and device tile scheduling.

Ownership. DistEnv is copied into the layer-0 runtime argument struct; layer 1 materializes the same fields.

Invariant. Slurm/torchrun rank assignment and NVSHMEM node-team rank must agree; the worktree checks nvshmem_team_my_pe(NVSHMEMX_TEAM_NODE) == local_rank.

Why this exists. The algorithm sends remote partials to the same-local-rank PE on the owner node.

Execution. Allocation is collective/symmetric; Flux wraps the raw pointer as a CUDA tensor without transferring ownership to PyTorch’s allocator.

Ownership. The symmetric heap owns storage; the tensor wrapper’s deleter synchronizes before nvshmem_free.

Invariant. Every participating PE makes compatible symmetric allocations in the same order and the heap is large enough.

Why this exists. Remote puts/gets require a remotely addressable symmetric object.

DIFF The patch deliberately uses cutlass::DeviceAllocation for CUDA stream wait/write addresses. Its comment records that PyTorch expandable segments can return a virtual address rejected by cuStreamWriteValue32_v2. This is a local implementation rationale, not a general ban on tensor-backed flags.

7. Put-with-signal: data plus its receipt

PAPER NVSHMEM exposes one-sided RMA and signaling over symmetric memory. A put-with-signal couples destination data delivery with a signal update. The receiver waits until its signal reaches the per-run value before consuming the slot.

A readiness flag answers “may the sender issue the put?” A receiver signal answers “has this sender’s put for this run arrived?” They are different addresses, on different sides of the transfer.

producer stores → system fence → release flag = 1 internode stream: wait(flag ≥ 1) → putmem_signal(..., signal = run_id) receiver stream: wait(signal ≥ run_id) → accumulate staging_recv
Stale-state prevention. Per-run flags and counters are memset to zero, an event orders that reset before auxiliary-stream waits, and monotonically increasing run_id values distinguish receiver signals across calls. Because each PE maintains its own counter, all PEs must invoke this operator in lock-step order so sender and receiver expect the same ID. Removing any one of those contracts can expose earlier state.

Glossary, snapshot, and primary sources

M / N / K
GEMM row, output-column, and reduction dimensions. In this code, M follows routed tokens; N follows output hidden width.
Symmetric slot
A fixed (node, split) region whose byte offset is identical in each PE’s corresponding allocation.
Release publication
A write whose observation also orders preceding writes at the required memory scope.
Run ID
A monotonically increasing signal value that prevents a prior invocation’s nonzero signal from satisfying a current wait.
