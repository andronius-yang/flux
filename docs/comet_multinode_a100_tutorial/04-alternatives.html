Alternatives: compare the optimization boundary

The local anchor: what exactly COMET couples

Three local mechanisms define the comparison target:

problem_schedules = get_sorted_problem_schedule(..., tile_M, 4);

Execution. Host orchestration turns ready-order M ranges into grouped problems.

Ownership / stream. Persistent symmetric input and per-call sorted indices feed the main-stream V3 launch.

Invariant. Each problem’s gather/scatter interval matches its scheduled M range.

Why. Make shared-input readiness part of GEMM work selection.

Execution. Reserved Hopper communication CTAs wait for one N split’s virtual completion target.

Ownership / stream. Compute and communication roles share one persistent launch and system barrier storage.

Invariant. Compute arrivals plus underfill compensation reach 132 − gather_rs_ctas.

Why. Keep long-latency I/O out of the TMA/MMA warp pipeline.

Execution. A100 CTAs perform local rings for each owner group and publish remote staging only after system visibility.

Ownership / stream. The gather-RS stream writes symmetric send slots; an auxiliary stream issues remote puts.

Invariant. The last local stage targets the same-local-rank owner PE, and flag observation implies slot readiness.

Why. Build a node hierarchy without adopting Hopper’s CTA execution model.

Comparison question. Does an alternative own this tile/N-split coupling, or a neighboring layer?

System-by-system notes

Megatron Core, Transformer Engine, and CUTLASS

PAPER Megatron offers MoE token dispatch/combine, grouped GEMM, batch-level all-to-all overlap, and shared-expert overlap. CUTLASS’s grouped scheduler persistently assigns tiles from multiple GEMM problems to bounded CTAs. Transformer Engine supplies optimized numerical kernels and recipes. Megatron guide; CUTLASS scheduler.

INFERENCE Verdict: these are the most credible baseline stack and direct building blocks. CUTLASS can implement COMET’s compute side, but its grouped scheduler alone does not create the readiness protocol or node hierarchy shown above.

FasterMoE / FastMoE

PAPER FasterMoE uses a performance model and expert shadowing: replicate a hot expert when moving its parameters is cheaper than routing all hot tokens remotely. That changes traffic and imbalance at a coarser decision boundary. PPoPP paper.

INFERENCE Verdict: complementary. Shadowing may reduce the bytes COMET must expose, but it is not shared-tensor tile decomposition.

Tutel

PAPER Tutel uses a unified layout to adapt parallelism and pipelines chunks of the MoE graph, including linear and 2DH all-to-all strategies. Its paper evaluates A100 systems at large scale. Tutel paper.

INFERENCE Verdict: a strong A100 runtime alternative at chunk/collective granularity. It does not imply the system-scope per-tile flags used by Flux.

MegaBlocks

PAPER MegaBlocks reformulates uneven expert work as block-sparse operations, avoiding padding or token dropping. Its distinguishing contribution is efficient dropless compute, not fused fine-grained transport. MegaBlocks paper.

INFERENCE Verdict: complementary. One could want MegaBlocks-style sparse expert compute and COMET-style communication exposure in the same broader stack, but integration is nontrivial.

DeepEP

PAPER DeepEP specializes dispatch/combine communication with high-throughput and low-latency modes and topology-aware NVLink/RDMA paths. Be version-specific: legacy V1 uses NVSHMEM; current V2 moved to NCCL Gin. Official repository; legacy V1 notes.

INFERENCE Verdict: a potential replacement for a transport/dispatch layer, not an automatic replacement for a fused expert GEMM scheduler. Its present public quick-start assumptions are not evidence for A100 support in this worktree.

MegaScale-MoE

PAPER MegaScale-MoE works at production training scale with communication-efficient parallel strategies, inter- and intra-operator overlap, and low-precision communication. It reports a 352B model on 1,440 Hopper GPUs. MegaScale-MoE paper.

INFERENCE Verdict: broader scope with some conceptual overlap. Neither system’s reported speedup can be transplanted to the other’s workload or hardware.

Likewise, nvshmem_fence orders but is not a general completion primitive; signal semantics apply to the corresponding put-with-signal. See the NCCL guide and NVSHMEM guide.

Evidence limits

  • PAPER COMET’s published results cover H800 and L20 configurations.
  • DIFF The multi-node A100 hierarchy is an uncommitted worktree patch.
  • INFERENCE No cross-project speedup is comparable without matching model, routing distribution, dtype, GPU, network, scale, training/inference mode, and baseline.
  • LOCAL NOTE No preserved log substantiates the local claim of two-node validation. This chapter reports no new measurement.

Glossary, snapshot, and primary bibliography

Optimization boundary
The compute, routing, transport, or scheduling layer a system directly owns.
Complement
A system that can improve a neighboring bottleneck without reproducing COMET’s fused schedule.
Overlap granularity
The smallest independently scheduled unit: batch, chunk, operator, N split, or tile.
