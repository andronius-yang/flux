1. The short answer

CODE Yes: F is the intermediate or FFN width of each expert. For expert e, the layer-0 mathematical problem has splits[e] token rows, so its unsharded shapes are:

input rows: [splits[e], H] layer-0 weight: [F, H] (PyTorch storage convention) layer-0 output: [splits[e], F] layer-1 weight: [H, F] layer-1 output: [splits[e], H]

Under FFN tensor parallelism of size P, the rank-local layer-0 weight is [F/P,H], and the rank-local intermediate is [splits[e],F/P]. The same splits[e] appears on all P ranks holding shards of expert e: the token-row dimension is preserved while those ranks divide the feature dimension.

Terminology correction. E, H, and F are chosen by the MoE model architecture/configuration. They are not chosen by the GPU architecture. Hopper versus Ampere selects a kernel family and execution strategy; it does not define how many experts the model has or the width of an expert.

2. COMET paper notation versus this tutorial and source

PAPER The COMET paper calls the token/row dimension M, token embedding width N, and expert feed-forward width K. This tutorial deliberately uses model-oriented names T, H, and F. Do not mechanically identify the tutorial’s H/F with every local GEMM variable named N/K; the latter can change with operand layout and layer.

INFERENCE Saying “experts per rank” without a qualifier is safe only in pure EP. Prefer:

  • local expert identities: always E/EP in this implementation;
  • complete experts physically held per rank: E/EP only when P=1;
  • expert weight shards per rank: E/EP, each with width F/P, when P>1.

Ownership / stream. This is host-side persistent operator metadata; no CUDA stream owns the calculation.

Invariant. W must be divisible by EP.

4. Two concrete rank grids

Hybrid example: W=8, EP=4, P=2, E=8

Each EP partition represents E/EP=2 expert identities. Each identity is split across two FFN-TP ranks, so no rank has a complete expert weight. If F=14336, each rank has F/P=7168 of the expert width.

EP 0experts 0–1, width 0–7167experts 0–1, width 7168–14335
EP 1experts 2–3, width 0–7167experts 2–3, width 7168–14335
EP 2experts 4–5, width 0–7167experts 4–5, width 7168–14335
EP 3experts 6–7, width 0–7167experts 6–7, width 7168–14335

If splits[3]=300, both ranks in EP partition 1 build a 300-row GEMM for expert 3. They use different F/P weight shards and produce different 300-by-7168 intermediate shards.

Your pure-EP case: W=8, EP=8, P=1, E=8

Each rank represents E/EP=1 expert identity, and because P=1, that rank holds the full F width. In this specific case, “one expert per rank” really does mean one complete expert weight per rank.

Execution. Global routing metadata is narrowed to the contiguous local expert-ID interval, while output columns use only F_local.

Ownership / stream. Helper kernels and GEMM launch are ordered on the caller’s CUDA stream; all-gather has its own orchestrated dependencies.

Invariant. Token counts come from global splits, but only the rank’s E/EP expert identities become local grouped-GEMM problems.

Why this code exists. EP removes nonlocal expert rows; FFN TP preserves those rows and divides their feature columns.

CODE Hopper layer 1 states the other half of the contract directly: local inputs and weights are partial to the current EP rank, while splits_cpu and routing_idx remain global.

Execution. Layer 1 forms problems for E/EP local experts, then uses global inverse routing to reconstruct token outputs.

Ownership / stream. Local input/weight tensors reside on the current GPU; routing metadata describes the whole fused group.

Invariant. sum(splits) matches the global routing length (absent dropped tokens), including when EP is enabled.

Why this code exists. Width-sharded ranks must agree on identical token rows so their partial feature results can be combined correctly.

6. How these dimensions enter Hopper’s communication–computation fusion

PAPER COMET describes pure EP as distributing intact experts and TP as partitioning expert weights along a hidden/FFN dimension; hybrid parallelism combines them and uses W=TP×EP. It then identifies the independent overlap dimension as token rows for layer 0 and output columns for layer 1.

CODE Hopper V3 realizes that decomposition by all-gathering the W input token shards, filtering routing to the current E/EP expert-ID interval, scheduling layer-0 tiles along arriving token rows, and computing only F/P feature columns on each FFN-TP rank. Its layer-1 path keeps the same local expert identities and global routing contract while splitting work along output columns.

Conclusion. The paper’s terminology and the original Hopper V3 implementation agree with your reasoning. The only refinement is linguistic: with P>1, say that a rank represents E/EP experts or owns E/EP expert shards—not that it owns that many complete experts.
