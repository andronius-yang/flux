Meta-design: why the repository has these seams

The file boundaries reflect legality, hardware, lifetime, and tuning. This chapter separates verified mechanisms from inferred rationale, then closes with a compatibility and correctness audit.

Data dependencies, not collective names, define legal decomposition

Layer 0: M is independent

A GEMM tile needs only the token rows in its source-rank interval. Per-source readiness plus an M-oriented problem schedule makes early compute legal.

Layer 1: N is independent

A result slice is legal after every expert GEMM problem has published that N slice. The top-k gather/reduction can then operate independently for that split.

INFERENCE Calling both operations “collectives” hides their useful partial orders. The source instead represents readiness at the same dimension used by the compute schedule.

docs/design.md:60–74 · HEAD · abridged paraphraserepository design note
Layer0 reschedules calculations along Dim-M.
Layer1 reschedules calculations along Dim-N.
Use horizontal fusion in kernels to isolate fine-grained I/O from computation.
On Hopper, persistent warp specialization uses producer TMA and consumer MMA roles.

Execution. Layer 0 schedules ready M ranges; layer 1 publishes and consumes ready N splits.

Ownership. The respective objects and kernels own readiness state for those dimensions.

Invariant. No tile moves before all of its dependencies are visible.

Why. The independent tensor dimension is the smallest unit that can move earlier without violating the MoE math.

Execution. Build-time generators register architecture/dtype families; runtime selection chooses within the matching family.

Ownership / stream. The registry owns implementation metadata; the selected operator later launches on the caller’s stream.

Invariant. Architecture, layout, dtype, and communication metadata match runtime arguments.

Why. Keep A100 and Hopper resource models explicit and tunable.

INFERENCE This duplication is deliberate specialization, not accidental redundancy. A single generic kernel would either discard Hopper mechanisms or force unavailable semantics onto A100.

Execution. Construction performs symmetric allocations, node-local peer mapping, pointer-array upload, event/stream creation, and capacity checks once.

Ownership. The object, not an individual forward, owns these resources.

Invariant. The object stays bound to the process group, CUDA device, topology, dtype/capacity contract, and serialized reuse assumptions with which it was created.

Why. Repeating collective allocation and peer discovery on every token batch would be expensive and error-prone.

POD arguments separate orchestration from generated kernels

CODE Torch-facing C++ validates tensors, manages streams and persistent state, then fills plain argument structs containing dimensions and raw pointers. Device adapters convert those structs into CUTLASS arguments. Generators register architecture/dtype/layout families; the registry selects hparams using runtime shape.

include/flux/args/moe_gather_rs.h:173–211 · DIFF · abridgedruntime ABI inside the process
struct TopKReduceGatherRSV2Arguments {
  void *output_ptr; int *splits; int *routing_idx;
  int rank, world_size, n_split;
  int **barrier; void **reduce_ptrs; int **tile_barrier_ptrs;
  int nnodes, node_idx, local_rank, local_world_size, staging_rows;
  void *staging_send; int *group_flags; int *group_counters;
};

Execution. Torch-facing orchestration fills the struct, registry selection chooses an adapter, and the adapter launches CUDA.

Ownership. The operator retains tensors and streams; the POD borrows raw addresses for one launch.

Invariant. Borrowed pointers remain valid and metadata matches topology, dtype, shape, and resources.

Why. Generated kernels receive compact values without depending on Torch or process-group types; tuning metadata remains part of resource assignment.

nnodes == 1 preserves the established path

DIFF V2 layer 0 keeps AllGatherOp only for one node. V2 layer 1 reduces to L=W and one owner group; local pointer arrays again span the full world. Multi-node-only allocations and host RMA are inactive.

INFERENCE This branch is a regression-containment boundary: the new topology must not tax the established same-node route or reinterpret remote pointers as locally accessible.

Transport belongs in launch policy

DIFF launch.sh selects libfabric/CXI only when Slurm reports multiple nodes, and retains legacy IB knobs for a single node. Generic CUDA kernels receive rank topology and symmetric pointers, not Perlmutter provider names.

Reason. Transport setup is site-specific; data-dependency and tile code should remain site-agnostic.

Verified code, inferred intent, and local claims

Compatibility / documentation

  • DIFF Inserting nnodes before n_split creates a positional-call compatibility hazard. Existing calls that passed later optional arguments positionally may be silently reinterpreted; keyword calls are safe.
  • DIFF TopkReduceScatterOp gains a pybind nnodes argument, but python/flux/cpp_mod.pyi:1010–1024 omits it.
  • DIFF examples/moe_flux_only.py propagates node count but still constructs V3 operators unconditionally; it is not an A100 example.
  • CODE docs/design.md has a “Fine-grained Remote I/O on Hopper” heading whose first explanatory paragraph begins with Ampere. The heading/content boundary is misleading.

Supported modes

  • DIFF Multi-node final accumulation dispatches FP16/BF16 only.
  • DIFF Triton/INT8, do_all_reduce, and use_read_mode remain single-node.
  • DIFF Gather-RS enforces m_full≤max_m; AG treats max_ntokens as capacity but has no explicit runtime bound check.

Hard deployment contracts

  • CODE Equal GPUs per node, node-contiguous global ranks, and NVSHMEM node-team local rank agreement.
  • CODE Token divisibility by world size/top-k, N % n_split == 0, N-split divisibility by tile/pack width, and staging capacity.
  • CODE Symmetric heap large enough for every live NVSHMEM allocation: AG input, send/receive staging, signals, and—when flux_create_tensor_list selects NVSHMEM—the reduce buffers and barriers; compatible collective allocation order.
  • CODE Compile-time ceilings are eight local GPUs, 32 world ranks, and eight nodes.
  • INFERENCE The operator is designed for serialized, lock-step reuse across PEs so process-local run IDs remain aligned. Concurrent or asymmetric calls are not documented as safe.

What is not evidence

  • LOCAL NOTE The untracked CLAUDE.md claims two-node validation, but no preserved test log appears in the snapshot.
  • DIFF Tracked test diffs propagate nnodes; they do not preserve a passing multi-node run.
  • INFERENCE The normal wrapper’s final collective barrier strongly protects outgoing staging reuse after the network exchange. Formal NBI source-lifetime semantics for direct low-level TopkReduceScatterOp.run use remain worth auditing.
  • LOCAL NOTE Perlmutter CXI settings are a local deployment recipe, not a portable A100-cluster guarantee.

Bottom line: the code clearly expresses an intended two-level algorithm and several strong ordering checks. This tutorial does not upgrade that into a correctness or performance claim without reproducible logs.
