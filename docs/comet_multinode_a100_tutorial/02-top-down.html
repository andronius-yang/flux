1. Plain PyTorch semantics

CODE The reference path makes the dependency chain explicit: communicate token shards, reorder them by expert, run expert GEMM0, apply GELU, run expert GEMM1, inverse-route the top-k copies, sum, and reduce-scatter.

Execution. The current stream executes the explicit reference stages in dependency order.

Ownership / stream. Input starts one token shard per world rank; expert weights are EP/FFN-TP-sharded; output returns one token shard.

Invariant. Routing metadata is identical across reference and fused paths.

Why. This is the semantic oracle, not the performance model.

2. Flux collapses the same graph into two calls

Execution. First call is all-gather + routing scatter + GEMM0. Second is GEMM1 + gather/top-k reduce + reduce-scatter.

Ownership / streams. Python supplies activations and weights; the objects retain staging, peer pointer arrays, barriers, events, and auxiliary streams.

Invariant. Fusing preserves the mathematical function and dependency order, subject to ordinary floating-point reassociation and rounding differences.

Why. Exposing the producer/consumer relation inside an operator enables tile-scale overlap that separate framework collectives cannot express.

3. SM90/V3 and pre-SM90/V2 diverge early

DIFF The architecture branch in examples/moe.py:160–186 keeps V3 for arch ≥ 90 and V2 otherwise. The new node count is orthogonal: it is propagated into DistEnvTPWithEP, V2 AG, and V2 gather-RS.

Do not say “A100 port of the Hopper kernel.” Hopper’s TMA/warp-specialized execution model is not present on SM80. This patch reuses orchestration ideas and preserves Ampere kernels.

5. Hopper layer 0: node-aware all-gather and M scheduling

src/moe_ag_scatter/ths_op/gemm_grouped_v3_ag_scatter.cc:158–224 · HEAD · abridgedmain + intra-node + inter-node streams
for (int node_idx = tp_env.node_idx, i = 0; i < tp_env.nnodes; ++i, ... ) {
  if (node_idx == tp_env.node_idx) { copy local shard; barrier_all_on_stream(main); }
  else { nvshmemx_getmem_on_stream(..., cp_stream_inter_node); ... }
  for (int local_rank = tp_env.local_rank, j = 0; j < tp_env.local_world_size; ++j, ...) {
    if (local_rank != tp_env.local_rank) nvshmemx_getmem_on_stream(..., cp_stream_intra_node);
    CUStreamWriteValue(cp_stream_intra_node, barrier_for(src_rank), 1, ...);
  }
}

Execution. Process current node first, then remote nodes; within each, distribute shards all-to-all among same-node GPUs and publish per-source readiness.

Ownership / stream. Full input is NVSHMEM symmetric; remote fetches use cp_stream_inter_node, local distribution and flag writes use cp_stream_intra_node.

Invariant. A consumer only reads a source rank’s rows after that source flag is written.

Why. Problem schedules order expert M tiles so ready token ranges can compute before the complete all-gather finishes.

CODE In gemm_grouped_v3_ag_scatter.cc:360–417, each scheduled expert interval becomes a grouped-GEMM problem with gather/scatter pointers. That is the concrete bridge from “M-oriented rescheduling” to input rows.

6. Hopper layer 1: horizontal CTA specialization

CODE V3 registers/tunes gather_rs_ctas. The persistent kernel reserves CTAs for communication while the remaining CTAs compute. Each N split has a barrier count equal to the number of compute CTAs.

src/moe_gather_rs/sm90_gemm_array_threadblock_specialized.hpp:1197–1216 · HEAD · abridgedcommunication CTA branch
int wait_flag = N_SMS_90 - GAHER_RS_N_CTAS;
for (int sid = split_start; sid < split_end; sid++) {
  Barrier::wait_eq(barrier, threadIdx.x, sid, wait_flag);
  for (int blk_m = dma_block_idx; blk_m < token_M_blk_count; blk_m += dma_blks)
    for (int blk_n = 0; blk_n < new_n_dim / GATHER_RS_BN; blk_n++)
      ep_gather_rs_impl_v3(..., sid, ...);

Execution. Compute CTAs publish completion for an N split; specialized CTAs wait and run gather/reduce-scatter work.

Ownership / sync. Both roles live in one persistent launch and coordinate through system-scope barriers.

Invariant. Compute arrivals plus block-0 compensation for an underfilled grid must reach the fixed target 132 − gather_rs_ctas.

Why. Isolate long-latency communication from Hopper’s efficient TMA/MMA warp pipeline.

Execution. The loop shape deliberately mirrors V3’s node-first fetch order.

Ownership / stream. The full input is symmetric; remote same-local-rank gets and intra-node fan-out use distinct auxiliary streams.

Invariant. Each per-source flag follows the transfer that makes that shard locally readable.

Why. Borrow the multi-node arrival protocol while leaving the registry-selected V2 grouped GEMM untouched.

Layer 1: hierarchy around the established V2 consumer

Execution. Main-stream GEMM and high-priority gather-RS overlap after a start event, then rejoin through a done event.

Ownership / stream. GEMM owns expert outputs and N-split producer barriers; the consumer stream owns local-ring progress and final output.

Invariant. Reset and reuse occur only after both streams rejoin and the process group synchronizes.

Why. Ampere obtains producer/consumer overlap with streams rather than Hopper CTA specialization.

DIFF The patched consumer replaces one world-size ring with one local-world-size ring per owner-node group. The last stage writes a local group directly to output and a remote group to symmetric staging_send. A separate stream performs put-with-signal; the main consumer stream waits for receiver signals and accumulates remote partials.

Execution. One launch.sh per node derives rendezvous and node rank from Slurm.

Ownership. libfabric/CXI variables are transport policy outside generic CUDA kernels.

Invariant. Slurm assigns node-contiguous global ranks and the symmetric heap fits AG plus both staging buffers/signals.

Why. Perlmutter’s Slingshot/CXI setup is deployment-specific and should not be generalized to arbitrary A100 clusters.

Two-node execution timeline

0 · Python
CPU: V2 operator selected; topology says two nodes. Persistent buffers already belong to the operator.
1 · Main stream
AG input: reset per-source flags; copy local shard into symmetric input; node barrier; record ready event.
2 · Inter-node stream
Remote node: fetch same-local-rank remote shard with NVSHMEM get; record fetch event.
3 · Intra-node stream
Local GPUs: peer-distribute each fetched shard; stream-write that global source rank’s readiness flag.
4 · Main stream
V2 layer-0 GEMM: waits on the most recently recorded remote-fetch completion event; tile fetchers gate individual row reads on source flags; the stream waits for full AG completion before reuse.
5 · Main + gather-RS streams
Layer 1: GEMM producer publishes N-split barriers; V2 consumer performs local rings for remote owner groups first.
6 · Consumer kernel
staging_send[g,sid]: system fence, all-block counter, release flag.
7 · Internode stream
Same-local-rank PE on node g: wait sender flag, then putmem_signal partial into its staging_recv[source,sid].
8 · Consumer stream
Owner node: wait each source signal ≥ run_id; accumulate FP16/BF16 packed partials into Python-visible [T/world,H] output.
9 · Main stream
Reuse boundary: wait consumer completion, synchronize group, reset local barriers/buffers for the next call.
INFERENCE This timeline is reconstructed from stream/event/flag code. It describes intended ordering, not a captured profiler trace.

Glossary, snapshot, and sources

Horizontal specialization
Different CTAs in one launch receive compute and communication roles.
Producer/consumer streams
Separate ordered CUDA queues linked by events and device-visible readiness state.
Reuse boundary
The synchronization point after which persistent staging may be overwritten by the next call.
